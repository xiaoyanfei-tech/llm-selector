# Packages

The goal is to help users start small, learn quickly, and avoid large AI spending before evidence.

Pricing is intentionally not listed here. The recommended path is to begin with the free CLI report, then request the smallest useful next step through GitHub issues.

## Free — Self-serve selection report

Best for:

- individual developers
- OPCs / one-person companies
- SMEs exploring first options

Includes:

- CLI-based model shortlist
- basic risk notes
- first PoC suggestions
- links to verification tools

Call to action:

- Run `python selector.py`
- Open an issue if you want feedback on the generated report

## Starter — OPC / Personal AI Stack

Best for:

- solo founders
- freelancers
- independent developers
- power users choosing Claude Code, Cursor, Cline, GLM, Qwen, DeepSeek, Kimi, GPT, or local models

Deliverables:

- personal workflow analysis
- recommended tool/model stack
- cost-control plan
- data-safety checklist
- 7-day adoption plan

Positioning:

- low friction
- fast decision
- no heavy consulting process

## Selection Memo — SME Model Choice

Best for:

- SMEs choosing first AI stack
- founders/IT managers needing a decision memo
- teams comparing hosted API, domestic providers, hybrid routing, and private deployment

Deliverables:

- executive summary
- top 3 model options
- risk register
- cost/deployment notes
- PoC recommendation
- verification checklist

Positioning:

- clear enough for a business decision
- technical enough for implementation planning
- not a long expensive consulting report

## AI Coding Stack Plan

Best for:

- engineering teams adopting AI coding assistants
- teams comparing Claude Code, Cursor, Cline, Continue, GLM, Claude, GPT, DeepSeek, Qwen, and gateway options

Deliverables:

- tool/model recommendation
- developer rollout plan
- endpoint verification plan
- cost and security checklist
- pilot success metrics

## PoC Launch Plan

Best for:

- teams ready to test one workflow in 2-4 weeks

Deliverables:

- PoC scope
- model/provider shortlist
- data handling rules
- success metrics
- rollout and stop/go criteria

## Market-entry philosophy

The project should win trust before it monetizes heavily:

1. give away useful first-pass reports
2. make the next step easy and low-risk
3. convert high-intent users into small paid engagements
4. use real PoC learnings to improve the open-source framework
5. move upmarket only after repeatable demand appears
